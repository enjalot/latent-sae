3 — The Dead Features Bug
Critical Discovery

Between v1 and v2 sweeps, we discovered three misconfigurations that explained the dead feature epidemic:


auxk_alpha = 0

The primary dead feature revival mechanism was disabled by default. Literature uses auxk_alpha = 1/32.

dead_threshold = 10M

Features only counted as "dead" after 10M tokens without firing — but our entire dataset was 10M. Threshold never reached.

multi_topk unscaled

EleutherAI uses L(k) + L(4k)/8 but we added L(4k) at full weight. 8x too strong, crowding out other features.

FVU: v2 architectures (auxk=1/32)
Dead %: v2 architectures (auxk=1/32)
v2 results: auxk helps TopK but doesn't save Gated. TopK dead features went from 0% (v1, small dict) to 35% (v2, with auxk competing against a 12K dictionary). Gated still collapses to 99% dead. JumpReLU unchanged — it ignores auxk by design. The real fix turned out to be smaller expansion factor.

8x expansion is the sweet spot. Same FVU as 32x (0.178 vs 0.200) with zero dead features and 4x fewer parameters. The embedding SAE literature (arXiv:2408.00657) predicted this — they used 2x-6x expansion for sentence embeddings. At 8x, all 3,072 features are utilized, the model trains faster, and every feature is interpretable.

multi_topk is harmful for embedding SAEs. It consistently produces 95%+ dead features across all configurations (v1 and v2, with and without auxk, with and without k-annealing). The mechanism that works for LLM-layer SAEs (training at 4x sparsity as auxiliary) doesn't transfer to the embedding setting. Don't use it.

LISTA doesn't help at this scale. All eta values produce worse FVU AND more dead features than vanilla TopK. Higher eta = worse results. The decoder-correlation approach is too blunt — similar decoder vectors don't necessarily mean features should suppress each other.
Why it fails: The on-the-fly correlation matrix treats all decoder similarity as redundancy, but some similar features capture genuinely different concepts at different activation levels. A learned S matrix (from PLANS.md) might do better, but at 8x expansion (3K features) the dictionary is small enough that lateral inhibition is unnecessary.
7 — Recommendations
Recommended Production Config
sae_type: topk
expansion_factor: 8        # 3,072 features at 384D, 6,144 at 768D
k: 64
auxk_alpha: 0.03125        # 1/32, standard dead feature revival
dead_feature_threshold: 50000
batch_size: 1024           # optimal for A10G
grad_acc_steps: 2          # effective batch = 2048
What works: TopK at 8x expansion with auxk=1/32. Zero dead features, 0.178 FVU, all features utilized, fast training.
What doesn't: multi_topk (kills features), Gated SAE (collapses), LISTA (hurts more than helps), expansion >16x (wasted capacity).
Still promising: JumpReLU for compact representations (0.032 FVU). May be useful for specific use cases where you want ~500 high-quality features rather than 3K diverse ones.
Next steps: Scale the recommended config to 100M samples on A10G (~$3, ~3 hours). Then compare probe accuracy on AG News / SST-2 / STS-B between TopK-8x and the published 32x model to verify that fewer, better-utilized features actually improve downstream task performance.
